Segurança em Uma Linha para IA Agêntica: Apresentando o SessionGuard

8 de março de 2026 · 8 min de leitura

Sistemas de IA agêntica — LLMs que chamam ferramentas, escrevem código e interagem com o mundo real — são a área de maior crescimento em segurança de IA. Uma única sessão de agente pode executar 50+ chamadas de ferramenta, e qualquer combinação delas pode constituir uma cadeia de ataque que nenhuma chamada individual dispararia.

Hoje estamos lançando o SessionGuard: uma camada de segurança unificada que combina log de auditoria, detecção de cadeia de ataque e inteligência de ameaças em uma única chamada guard.check(). Uma linha de código oferece proteção em tempo real, trilha de auditoria completa e exportação pronta para SIEM.

O Problema: Segurança Agêntica é Multidimensional

As ferramentas de segurança existentes resolvem partes do quebra-cabeça:

  • Scanners de entrada capturam prompt injection em mensagens individuais
  • Validadores de uso de ferramentas bloqueiam comandos perigosos um de cada vez
  • Loggers de auditoria registram o que aconteceu depois do fato

Mas ataques agênticos são sequências. Um atacante não simplesmente executa rm -rf / — primeiro executa whoami (reconhecimento), depois cat .env (acesso a credenciais), depois curl -X POST (exfiltração). Cada etapa parece inofensiva isoladamente.

O SessionGuard resolve isso combinando três sistemas em uma única chamada:

ComponenteO Que FazPor Que Importa
SessionAuditLog encadeado por hash, à prova de adulteraçãoConformidade com SOC 2, ISO 27001
AttackChainDetectorDetecção de sequências de ataque em múltiplas etapasCaptura o que scanners de chamada única não detectam
ThreatFeedCorrespondência de padrões MITRE ATLASDetecção de técnicas de ataque conhecidas

Uso: Uma Linha

from sentinel.session_guard import SessionGuard

guard = SessionGuard(
    session_id="session-123",
    user_id="user@org.com",
    block_on="critical",  # Block anything >= critical risk
)

# In your agent loop:
for tool_call in agent.tool_calls():
    verdict = guard.check(tool_call.name, tool_call.arguments)

    if not verdict.allowed:
        print(f"BLOCKED: {verdict.block_reason}")
        continue

    if verdict.warnings:
        print(f"WARNING: {verdict.warnings}")

    # Execute the tool call
    result = execute_tool(tool_call)

O Que É Verificado

Cada chamada guard.check() executa quatro camadas de análise:

1. Detecção de Comandos Destrutivos

Padrões regex compilados capturam comandos perigosos instantaneamente:

guard.check("bash", {"command": "rm -rf /"})
# GuardVerdict(allowed=False, risk="critical",
#   block_reason="destructive_command: ...")

Captura: rm -rf, fork bombs, dd para disco, chmod 777 /, curl | bash, e mais.

2. Acesso a Arquivos Sensíveis

guard.check("read_file", {"path": ".env"})
# GuardVerdict(allowed=True, risk="high",
#   warnings=["Sensitive file access: .env"])

Monitora o acesso a: .env, .aws/credentials, .ssh/id_rsa, /etc/shadow, .kube/config, e mais 10+ caminhos sensíveis.

3. Correspondência com Inteligência de Ameaças

guard.check("bash", {"command": "echo 'ignore all previous instructions'"})
# GuardVerdict(threat_matches=["[critical] Direct Instruction Override"])

Faz correspondência com 27 indicadores alinhados ao MITRE ATLAS em prompt injection, jailbreaks, exfiltração de dados, escalonamento de privilégios e técnicas de evasão.

4. Detecção de Cadeia de Ataque

guard.check("bash", {"command": "whoami"})          # reconnaissance
guard.check("read_file", {"path": ".env"})            # credential access
guard.check("bash", {"command": "curl -X POST ..."})  # exfiltration
# GuardVerdict(chains_detected=["recon_credential_exfiltrate"], risk="critical")

Detecta 6 cadeias de ataque em múltiplas etapas, incluindo recon→credential→exfiltrate, escalation→destruction, e sequências de envenenamento de contexto.

Regras Personalizadas

Adicione regras de bloqueio específicas do projeto:

guard = SessionGuard(
    block_on="high",
    custom_rules=[
        # Block npm publish in CI
        lambda tool, args: "no_publish"
            if "npm publish" in args.get("command", "")
            else None,
        # Block writes to production config
        lambda tool, args: "prod_config_protected"
            if "/prod/" in args.get("path", "")
            else None,
    ],
)

Exportação para SIEM

Cada verificação é registrada com encadeamento de hash SHA-256. Exporte a trilha completa ao final da sessão:

report = guard.export()

# report = {
#   "session_id": "session-123",
#   "user_id": "user@org.com",
#   "integrity_verified": True,
#   "summary": {
#     "total_calls": 47,
#     "blocked_calls": 2,
#     "risk_level": "critical",
#     "tool_counts": {"bash": 20, "read_file": 15, ...},
#     "finding_categories": {"credential_access": 3, ...}
#   },
#   "active_chains": [
#     {"name": "recon_credential_exfiltrate", "severity": "critical", ...}
#   ],
#   "entries": [...]  # Every check, hash-chained
# }

Envie o JSON para Splunk, Datadog, Elastic, ou qualquer SIEM. O encadeamento de hash permite provar que nenhuma entrada foi adulterada — essencial para auditorias de conformidade SOC 2 e ISO 27001.

SDK TypeScript

Paridade total no SDK TypeScript:

import { SessionGuard } from '@sentinel-ai/sdk';

const guard = new SessionGuard({
  sessionId: 'session-123',
  userId: 'user@org.com',
  blockOn: 'critical',
});

const verdict = guard.check('bash', { command: 'ls src/' });
console.log(verdict.safe); // true

Arquitetura

Princípio de design: o SessionGuard é um coordenador, não um novo scanner. Ele combina componentes já existentes e testados em batalha (cada um com seus próprios conjuntos de testes) em uma API unificada. Isso significa que você obtém a confiabilidade de três sistemas comprovados com a simplicidade de uma única chamada.

Internamente, cada chamada check():

  1. Verifica contra padrões compilados de comandos destrutivos (sub-microssegundo)
  2. Varre em busca de acesso a arquivos sensíveis (correspondência de string O(1))
  3. Faz correspondência com o ThreatFeed (27 padrões regex compilados)
  4. Registra no AttackChainDetector (classificação de estágio em janela deslizante)
  5. Aplica regras personalizadas (lambdas definidas pelo usuário)
  6. Toma a decisão de bloquear/permitir com base no limiar
  7. Registra no SessionAudit com encadeamento de hash

Sobrecarga total por verificação: <0,1ms. Zero chamadas de rede. Zero dependências além de regex.
