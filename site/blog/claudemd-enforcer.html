Transformando Regras do CLAUDE.md em Guardrails Determinísticos

8 de março de 2026 · 7 min de leitura

Todo projeto do Claude Code tem um arquivo CLAUDE.md com regras como "Nunca use rm -rf" ou "Não acesse .env." Essas regras são críticas para a segurança — mas são apenas sugestões. O modelo as lê, as entende e geralmente as segue. Mas "geralmente" não é bom o suficiente quando o que está em jogo são dados de produção, credenciais ou infraestrutura.

Hoje estamos lançando o ClaudeMdEnforcer: uma ferramenta que lê seu CLAUDE.md, extrai as regras de proibição e as converte em verificações determinísticas que realmente bloqueiam violações antes que sejam executadas.

O Problema: Regras Sem Aplicação

Considere um CLAUDE.md típico:

# Project Rules
- Never use `rm -rf`
- Do not access `.env` or `.aws/credentials`
- Never run `git push --force`
- Do not modify files in `tests/` without asking

Essas regras existem na janela de contexto do modelo. O modelo as vê e geralmente cumpre. Mas há modos de falha:

Quando as regras do CLAUDE.md falham:
  • Overflow da janela de contexto — sessões longas empurram as instruções do CLAUDE.md para fora da janela de contexto
  • Prompt injection — o payload de um atacante em um arquivo sobrepõe as instruções do CLAUDE.md
  • Jailbreaks — prompts adversariais convencem o modelo a ignorar suas instruções
  • Atualizações de modelo — uma nova versão do modelo pode interpretar as regras de forma diferente
  • Ambiguidade — "Não modifique tests/" — renomear um arquivo de teste conta?

A questão fundamental: as regras do CLAUDE.md são aplicadas pelo mesmo sistema que elas tentam restringir. É como pedir para um programa aplicar sua própria política de segurança sem um kernel de SO.

A Solução: Aplicação Determinística

O ClaudeMdEnforcer adiciona uma camada de aplicação em nível de kernel. Ele faz o parsing do seu CLAUDE.md, extrai regras de proibição usando pattern matching de NLP e gera verificações determinísticas que rodam fora do controle do modelo:

from sentinel import ClaudeMdEnforcer

# Faz o parsing do seu CLAUDE.md
enforcer = ClaudeMdEnforcer.from_file("CLAUDE.md")

# Verifica uma tool call
verdict = enforcer.check("bash", {"command": "rm -rf /tmp"})
print(verdict.allowed)       # False
print(verdict.violated_rules) # ["Never use `rm -rf`"]

# Veja quais regras foram extraídas
print(enforcer.summary())
# Extracted 4 enforceable rule(s) from CLAUDE.md:
#   Blocked Commands (2):
#     L2 rm -rf  — Never use `rm -rf`
#     L4 git push --force  — Never run `git push --force`
#   Blocked Paths (2):
#     L3 .env  — Do not access `.env` or `.aws/credentials`
#     L5 tests/  — Do not modify files in `tests/`

Como Funciona

O ClaudeMdEnforcer usa um pipeline de três estágios:

1. Detecção de Proibições

O parser identifica regras de proibição usando padrões linguísticos:

PadrãoExemplo
never X"Never use rm -rf"
do not X"Do not access .env"
must not X"You must not run wget"
X is forbidden"curl is forbidden"
avoid X"Avoid modifying the database"

Ele lida com formatação Markdown: itens de lista, conteúdo entre crases, negrito/itálico e parágrafos de texto corrido. Documentação comum (como "o usuário não deve ver erros") é ignorada — só regras acionáveis que restringem ferramentas são extraídas.

2. Extração de Comando e Caminho

Uma vez identificada uma proibição, o parser extrai o comando, caminho ou ferramenta específicos que estão sendo bloqueados:

  • "Never use `rm -rf`" → blocked_command: rm -rf
  • "Do not modify files in `tests/`" → blocked_path: tests/
  • "curl is forbidden" → blocked_tool: curl
  • "Never run `git push --force`" → blocked_command: git push --force

3. Aplicação em Tempo de Execução

Cada regra extraída vira uma regex compilada que é testada contra cada tool call. A verificação roda em microssegundos — sem inferência de LLM, sem chamadas de API, sem latência.

Integração com Hooks do Claude Code

O recurso matador: o ClaudeMdEnforcer se conecta diretamente ao sistema de hooks PreToolUse do Claude Code. Toda tool call é verificada antes da execução:

# Instala o sentinel e configura os hooks:
pip install sentinel-guardrails
sentinel init

# Ou adicione manualmente ao .claude/settings.json:
{
  "hooks": {
    "PreToolUse": [{
      "matcher": ".*",
      "hooks": [{
        "type": "command",
        "command": "sentinel hook"
      }]
    }]
  }
}

Agora toda chamada de Bash, Write e Edit é verificada contra suas regras do CLAUDE.md. Se uma regra for violada, a tool call é bloqueada com exit code 2 e uma mensagem clara explicando o motivo.

Uso via CLI

O comando sentinel enforce te dá acesso completo pelo terminal:

# Extrai e exibe as regras
sentinel enforce --file CLAUDE.md --show-rules

# Verifica uma tool call específica
sentinel enforce --file CLAUDE.md --tool bash --command "rm -rf /tmp"
# Saída: [BLOCKED] bash — Rule violated: Never use `rm -rf`

# Exporta como uma guard policy YAML
sentinel enforce --file CLAUDE.md --export-policy > policy.yaml

# Saída em JSON para integração com CI/CD
sentinel enforce --file CLAUDE.md --export-policy --format json

Geração de Política

O ClaudeMdEnforcer pode exportar suas regras extraídas como uma GuardPolicy — um objeto de política declarativo que funciona com o SessionGuard para aplicação completa em nível de sessão:

enforcer = ClaudeMdEnforcer.from_file("CLAUDE.md")

# Converte para uma guard policy
policy = enforcer.to_guard_policy()

# Cria um session guard com a política
guard = policy.create_guard(session_id="agent-session-1")

# Agora toda tool call é verificada
verdict = guard.check("bash", {"command": "rm -rf /"})
# Blocked by CLAUDE.md rule AND by built-in dangerous command detection

Isso te dá defesa em camadas: regras do CLAUDE.md mais escaneamento de segurança embutido, tudo em uma única chamada check().

SDK em TypeScript

O ClaudeMdEnforcer também está disponível no SDK TypeScript:

import { ClaudeMdEnforcer } from '@sentinel-ai/sdk';

const enforcer = ClaudeMdEnforcer.fromText(claudeMdContent);
const verdict = enforcer.check('bash', { command: 'rm -rf /tmp' });

if (!verdict.allowed) {
  console.log('Blocked:', verdict.violatedRules);
}

// Export to policy
const policy = enforcer.toGuardPolicy();

O Que É Extraído (e o Que Não É)

Extraído como regras aplicáveis:
  • "Never use rm -rf" → bloqueia qualquer comando bash contendo rm -rf
  • "Do not access .env" → bloqueia leituras/escritas de arquivo em .env
  • "curl is forbidden" → bloqueia todas as tool calls de curl
  • "Blocked commands: mkfs, dd, fdisk" → bloqueia cada comando listado
Não extraído (por design):
  • "The application should handle errors gracefully" — não é uma restrição de ferramenta
  • "Use TypeScript for new files" — preferência afirmativa, não uma proibição
  • "Follow PEP 8 style" — orientação de estilo, não uma regra de segurança

O parser foca deliberadamente em proibições relevantes para segurança — regras que restringem o que as ferramentas podem fazer. Preferências de estilo e orientação arquitetural permanecem no contexto do modelo, onde pertencem.

Defesa em Profundidade

O ClaudeMdEnforcer é uma camada de uma estratégia de defesa em profundidade:

CamadaO QuêAplicado Por
1Regras do CLAUDE.mdContexto do modelo (soft)
2ClaudeMdEnforcerRegex determinística (hard)
3SessionGuardDetecção de cadeia de ataque (smart)
4GuardPolicyPolítica YAML declarativa (configurável)

A camada 1 cobre 99% dos casos. As camadas 2 a 4 capturam o 1% que mais importa.

pip install sentinel-guardrails
